## 产品概述

### 阶段一：CSP 导购端 7 项 UI/逻辑调整

1. **删除"核心"标识**：移除所有竞品卡片上的"核心"/"核心竞品"标签
2. **每日数据上报改为二级页面**：从 TabBar 移除，改为首页 navigateTo 跳转，解决底部按钮被遮挡问题
3. **月报上报时间可配置**：4月数据在5月1-5日可上报，页面增加时间提示
4. **删除每日历史"已通过"状态**：统计卡片移除已通过列，筛选 Tab 移除已通过选项
5. **月报增加上报时间提示**：显示"下个月几号到几号可上报"
6. **简化待办列表**：日数据只显示已提交/未提交，月数据保留提交/通过/草稿
7. **我的页面文案调整**：日报→日竞品数据上报，月报→月竞品数据上报

### 阶段二：AC 端开发（主管审核端）

1. **AC 端首页**：待审批统计、异常提醒、任务切换（UI 与 BA 端对齐）
2. **月报审核工作台**：查看待审核月报、通过/驳回/退回操作
3. **数据打回功能**：可将异常数据打回，门店重新填写
4. **升级规则**：月结后3天未审核自动升级至区域经理
5. **数据报表模块**：绩效监控，数据可视化

### 阶段三：管理后台（系统管理端）- 完整功能

1. **数据管理**：日报/月报数据查看、筛选、导出
2. **数据导入**：支持 Excel 批量导入日报/月报数据
3. **数据修正**：管理员可修改门店提交的数据、打回重填
4. **配置管理**：品牌/商场/柜台管理、上报时间配置
5. **用户管理**：用户 CRUD，角色分配，Digital Hub 同步
6. **权限控制**：BA(仅指定柜台)、AC(仅指定区域)、经理(全区域)、Admin(全系统)
7. **审计日志**：所有数据变更记录，保留2年

### AC3 - 月报驳回功能

#### 驳回流程

```
AC 端                        BA 导购端
   │                           │
   │───── 审核驳回 ───────────▶│
   │     (填写原因)             │
   │                           │── 收到驳回通知
   │                           │── 待办列表显示"待修改"
   │                           │── 修改后重新提交
   │◀──── 重新提交 ───────────│
   │                           │
```

#### AC 端驳回功能

- **驳回入口**：审核详情页操作按钮
- **驳回弹窗**：选择驳回原因（预设原因 + 自定义）
- **预设原因**：
- 数据异常（波动过大）
- 数据缺失
- 格式错误
- 其他（自定义）
- **驳回记录**：保留驳回历史，便于追溯

#### BA 导购端接收驳回

- **待办列表**：增加"待修改"状态的驳回数据
- **驳回通知**：推送消息提醒 + 首页异常提示
- **修改入口**：点击驳回数据进入修改页面
- **重新提交**：修改后提交，状态变更为"已提交"

## Implementation Approach

### 关键设计决策

1. **TabBar 页面 vs 二级页面**：

- TabBar 页面使用 `switchTab` 跳转
- 二级页面使用 `navigateTo` 跳转，有返回按钮

2. **后台可配置时间**：

- 在 `constants.ts` 定义配置常量
- 后续可改为接口动态获取

3. **状态简化**：

- 每日数据：只有 `submitted`（已提交）和无状态（未提交）
- 月度数据：保留 `draft`/`submitted`/`approved` 三种状态

## Implementation Notes

1. **回归风险**：修改均为 UI 文本和筛选逻辑，风险较低
2. **注意**：删除核心标识时保留 type 字段（后端数据需要）
3. **TabBar 切换**：首页使用自定义 TabBar，切换逻辑不变
